### Concurrent Contents-API commits clobbered a log line instead of merging (lost-update race)

- **Issue:** while both `applicant-triage` and `chess-arbiter` were working around the missing-git-identity gap by committing via the GitHub Contents API (`gh api -X PUT .../contents/...`), `chess-arbiter`'s commit (`f81be0d228`, 2026-07-06T21:52:14Z) *replaced* the just-added `applicant-triage: 0 processed (0 auto-admitted, 0 queued, 0 rejected)` line (from `applicant-triage`'s commit `b1c9d6b2ba`) with its own `## chess-arbiter @ ...` line, instead of appending after it — the applicant-triage line was silently lost from `memory/logs/2026-07-06.md`.
- **Observed:** confirmed via `git show f81be0d228 -- memory/logs/2026-07-06.md`: the diff is a straight replace of the last line, not an append. This is a classic lost-update race — the API's `sha` precondition only protects against writing over a *stale* blob, but if the actor computing the new file content re-reads a version that predates the sibling's commit (or simply appends without re-fetching immediately before the PUT), one commit's addition still gets dropped even though the SHA check passed at each individual PUT. This same anomalies.md file hit the identical race twice more while writing *this entry* — two other concurrent skills (`feed-rollup`, and an earlier retry of this note itself) each landed their own Contents-API commit to this file in the same few-minute window, requiring a re-fetch-and-merge each time before this note could land.
- **Risk:** any concurrent admin-skill runs that both fall back to Contents-API commits (as multiple skills now do, per the missing-git-identity entries above) can silently drop each other's log lines or, worse, other content — this same mechanism could clobber a real moderation log entry, not just a routine log line, if two skills act in the same window. The frequency observed this cycle (3 collisions on 2 different files within about 15 minutes) suggests this is not a rare edge case right now — it's the default outcome whenever two skills are both mid-cycle.
- **Corrective action taken:** re-fetched current remote content and re-appended the missing `applicant-triage` line (restore + this cycle's fresh entry) in commit `006f1aa4fa`; re-fetched and merged this anomalies.md note itself twice more before it landed cleanly.
- **Status:** Founder/infra review needed — the missing-git-identity gap (see entries above) is pushing multiple skills onto a commit path (Contents API) that isn't safe under concurrency the way a real `git commit` + `git push` (with merge/rebase-on-conflict) would be. Fixing the git-identity gap would remove the need for this workaround entirely; short of that, skills using the Contents-API fallback should re-fetch the current sha and content immediately before each PUT (minimizing but not eliminating the race) and diff-check that their own prior addition wasn't dropped.
- **First noted:** 2026-07-06, during this `applicant-triage` run.
- **Corroboration (2026-07-06T22:15:00Z, `collab-sub-enforcer` run):** hit the same collision — `sub-create`'s commit (`b83c79a226`) landed on `memory/logs/2026-07-06.md` between this session's local edit and push. This session *did* have a usable local git identity path this time (per-command `git -c user.name=... -c user.email=...` worked, unlike the fully-missing-identity runs above), so it went through plain `git commit` + `git push` instead of the Contents API. Result: `git push` was rejected (non-fast-forward), `git rebase origin/main` surfaced a real merge conflict on the shared log file instead of silently clobbering, and the conflict was resolved by hand (keeping both lines, reordered chronologically) before pushing.
